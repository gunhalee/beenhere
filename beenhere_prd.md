## 1. 제품 개요

### 1.1 제품 정의

beenhere는 사용자가 물리적 장소에 글을 남기고, 그 장소를 방문한 사람들이 글을 읽으며 마음에 드는 글을 수집하는 위치 기반 글 서비스다.

### 1.2 핵심 컨셉

"장소에 가면 그곳의 이야기가 있고, 이야기를 따라 사람이 있다."

### 1.3 핵심 원칙

- 글은 작성된 장소에 고정된다. 이전되지 않는다.
- 수집은 글을 내 프로필에 모이는 행위이며, 그 글이 해당 장소에서 더 잘 보이게 만드는 신호다.
- 댓글, 팔로우, 알림, 알고리즘은 존재하지 않는다.
- 서비스가 사용자를 부르지 않는다. 사용자가 스스로 찾아온다.
- 탐색은 글과 프로필의 연쇄를 통해 발생한다.
- 프로필은 기본적으로 열려 있으며, 차단은 안전을 위한 예외적 장치로만 존재한다.

## 2. 배경과 문제 정의

### 2.1 우리가 해결하려는 문제

기존 소셜 서비스는 사람 중심이다. 누가 누구를 팔로우하는지, 알고리즘이 무엇을 밀어주는지가 경험의 중심이 된다. 그 결과 장소는 배경으로 밀려나고, 글은 맥락을 잃는다.

beenhere는 반대로 장소를 먼저 세운다. 특정 위치에 남겨진 글을 통해 사용자는 그 장소의 분위기와 흔적을 만나고, 그 글을 쓴 사람을 뒤늦게 발견한다.

### 2.2 제품 가설

- 사람들은 특정 장소에서만 읽히는 글에 더 강한 맥락과 정서를 느낀다.
- 누군가의 프로필을 먼저 찾기보다, 글을 통해 우연히 사람을 발견하는 경험은 별도의 매력을 만든다.
- 댓글, 팔로우, 알림이 없는 느린 상호작용 구조는 피로도를 낮추고 서비스의 문학적 정체성을 강화한다.
- 수집이 곧 로컬 신호인 구조는 알고리즘 없이도 동네마다 고유한 피드를 만든다.

### 2.3 왜 지금 필요한가

- 지역 기반 서비스는 대체로 거래, 정보, 생활 편의 중심이며 정서적 기록이 약하다.
- 기존 SNS는 장소 특유의 감각보다 관계망과 반응 수치에 과도하게 최적화되어 있다.
- beenhere는 장소성, 우연한 발견, 느린 상호작용이라는 세 가지 감각을 결합해 새로운 장르의 글 경험을 제안한다.

## 3. 목표와 비목표

### 3.1 MVP 목표

- 사용자가 특정 장소에 글을 남기고 주변에서 그 글이 읽히게 한다.
- 사용자가 다른 사람의 글을 수집하며 자신의 프로필에 모을 수 있게 한다.
- 사용자가 글을 통해 타인의 프로필로 들어가 탐색을 이어가게 만든다.
- 수집 신호가 로컬 신호로 작동하여, 동네마다 유의미한 피드 외형이 형성되는지 검증한다.

### 3.2 비즈니스/제품 검증 질문

- 사용자는 "위치에 묶인 글"이라는 제약을 불편함보다 매력으로 받아들이는가?
- 댓글/팔로우/알림이 없어도 반복 방문과 탐색이 발생하는가?
- 수집 신호만으로 동네마다 유의미한 피드 외형이 형성되는가?
- 글 → 작성자 프로필 → 다른 글 → 다른 사람으로 이어지는 탐색이 자연스럽게 발생하는가?

### 3.3 MVP 비목표

- 대화형 커뮤니티를 만드는 것
- 크리에이터 팔로잉 네트워크를 만드는 것
- 글의 지리적 이전 및 재공유 구조를 만드는 것
- 실시간 리텐션 장치를 촘촘히 설계하는 것
- 개인화 추천 피드를 만드는 것
- 전파 경로를 시각화하는 것

## 4. 타깃 사용자

### 4.1 1차 타깃

- 장소에 감정을 이입하는 사람
- 산책, 이동, 여행 중 우연한 발견을 좋아하는 사람
- 짧은 글 또는 메모 형태로 흔적을 남기고 싶은 사람
- 과도한 사회적 상호작용 없이 타인의 취향과 문장을 엿보고 싶은 사람

### 4.2 핵심 사용자 유형

**남기는 사람**

특정 장소에서 떠오른 생각, 감정, 장면을 글로 남기고 싶은 사용자.

**수집하는 사람**

마음에 든 글을 자신의 프로필에 모으고 싶은 사용자.

**따라가는 사람**

글을 계기로 작성자의 프로필을 탐색하며 사람을 발견하는 사용자.

### 4.3 JTBD

- 내가 서 있는 장소에 맞는 글을 읽고 싶다.
- 어떤 장소에서 떠오른 감정을 그 장소에 그대로 남기고 싶다.
- 마음에 든 글을 수집하여 나중에 다시 꺼내보고 싶다.
- 반응을 주고받기보다, 글의 흔적을 따라 사람을 조용히 발견하고 싶다.

## 5. 사용자 경험 흐름

### 5.1 발견

앱을 열면 사용자의 현재 위치에서 가까운 순서로 글이 피드에 표시된다. 같은 거리권 안에서는 해당 글의 작성 위치 기준 최근 수집 활동이 활발한 글이 우선 노출된다. 수집이 이어지지 않는 글은 자연스럽게 후순위로 밀린다.

### 5.2 탐색

작성자의 프로필에 진입한다. 그 사람이 작성한 글 피드와 수집한 글 피드를 보고, 거기서 또 다른 사람의 프로필로 이어진다. 탐색에 깊이 제한은 없다.

### 5.3 수집

마음에 드는 글을 수집하면 내 프로필에 추가된다. 수집은 해당 글의 작성 위치 기준 로컬 수집 수로 집계되어 그 장소 피드에서의 노출 우선순위에 반영된다.

### 5.4 확인

자신이 작성한 글의 수집 수와 수집한 사람 목록을 확인한다. 수집한 사람의 프로필에 들어가 어떤 사람이 자신의 글에 공명했는지 살펴본다. 이 정보는 작성자 본인만 볼 수 있다.

### 5.5 반복 방문

알림이 없기 때문에 사용자는 앱을 열어 직접 근처 글의 변화를 확인한다. 반복 방문의 동기는 "누가 나를 찾았는가"보다 "이 장소에는 지금 어떤 글이 떠 있는가"에 가깝다.

## 6. MVP 범위

### 6.1 포함 범위

- 디바이스 ID 기반 서비스 이용 (가입 없이 즉시 사용 가능)
- 랜덤 닉네임 기반 프로필 생성
- 위치 권한 요청 및 현재 위치 기반 피드 조회
- 동적 탐색 거리 (피드 내 작성글 수합이 10 미만일 경우 자동 확장)
- 글 작성
- 글 수집 및 수집 취소
- 프로필 탐색
- 사용자 차단
- 작성자 전용 수집한 사람 목록 확인
- 거리 기반 + 로컬 수집 수 기반 피드 정렬

### 6.2 제외 범위

- 댓글
- 팔로우
- 푸시 알림
- 알고리즘 추천
- DM
- 사진/영상 등 미디어 첨부
- 글의 지리적 이전 및 재공유
- 전파 경로 시각화
- 프로필 비공개

### 6.3 운영 필수 범위

아래는 핵심 경험에는 드러나지 않더라도 서비스 운영상 필요한 최소 범위다.

- 신고 접수 및 운영자 검토
- 운영자에 의한 게시물 비노출/삭제
- 스팸 및 반복 도배 제한
- 법적 요청 대응을 위한 기본 로그 보존

## 7. 기능 명세

### 7.1 계정 및 온보딩

| 항목 | 내용 |
|-|-|
| 기본 이용 방식 | 가입 없이 디바이스 ID 기반으로 모든 서비스 이용이 가능하다 |
| 닉네임 발급 방식 | 최초 진입 시 공개 이름은 랜덤 닉네임으로 자동 생성된다 |
| 닉네임 재생성 | 사용자는 랜덤 닉네임을 재생성할 수 있다 |
| 닉네임 적용 규칙 | 계정에는 항상 하나의 활성 닉네임만 존재하며, 작성한 글과 프로필에는 모두 동일한 닉네임이 표시된다 |
| 프로필 구성 | 프로필은 활성 닉네임, 작성한 글 피드, 수집한 글 피드로만 구성된다 |
| 프로필 비포함 요소 | 소개 문구, 상태 메시지, 자기소개, 링크 등 추가 프로필 요소는 제공하지 않는다 |
| 위치 권한 | 피드 조회와 글 작성을 위해 위치 권한이 필요하다 |
| 권한 거부 시 | 위치 기반 피드는 제한되며, 자신의 프로필 열람 정도만 가능하게 설계한다 |

### 7.2 글 작성 및 노출

| 항목 | 내용 |
|-|-|
| 작성 조건 | 사용자는 현재 위치에서 글을 작성할 수 있다 |
| 콘텐츠 형식 | MVP에서는 텍스트만 지원한다 |
| 글 길이 제한 | 글 본문은 최대 300자까지 작성할 수 있다 |
| 좌표 고정 | 글은 작성 장소의 좌표에 고정된다. 이후 이동하지 않는다 |
| 노출 범위 | 고정된 좌표로부터 탐색 거리 이내에 접속한 다른 사용자의 피드에 노출된다 |
| 자기 글 열람 | 자신이 작성한 글은 위치와 무관하게 항상 자신의 프로필에서 볼 수 있다 |

### 7.3 수집

| 항목 | 내용 |
|-|-|
| 수집 가능 위치 | 피드에서 또는 타인의 프로필에서 발견한 글을 수집할 수 있다 |
| 수집의 효과 | 수집한 글은 내 프로필에 추가된다. 위치와 무관하게 항상 열람 가능하다 |
| 로컬 신호 | 수집은 해당 글의 작성 위치 기준 10km 이내에서 발생한 수집으로 집계되어 그 장소 피드의 노출 우선순위에 반영된다 |
| 수집 수 노출 | 수집 수는 글을 보는 모든 사람에게 노출된다 |
| 수집 취소 | 수집을 취소할 수 있으며, 취소 시 프로필에서 제거되고 로컬 수집 수에서 차감된다 |
| 재수집 | 수집 취소 후 동일한 글을 다시 수집할 수 있다 |
| 중복 수집 | 한 사용자는 동일한 글을 동시에 한 번만 수집 상태로 유지할 수 있다 |

### 7.4 글의 노출

| 항목 | 내용 |
|-|-|
| 기본 노출 | 글은 작성 후 지리적 피드에 노출된다. 별도의 만료 기간 없이 삭제되지 않는다 |
| 노출 우선순위 | 수집 활동이 최근에 이루어진 글일수록 피드에서 우선 노출된다. 수집이 이어지지 않는 글은 자연스럽게 후순위로 밀린다 |
| 프로필 보존 | 작성자와 수집한 사람의 프로필에서는 항상 볼 수 있다 |
| 재활성화 | 오래되어 후순위로 밀린 글이 다시 수집되면, 수집 활동 기준으로 노출 우선순위가 높아진다 |

### 7.5 피드

| 항목 | 내용 |
|-|-|
| 탐색 거리 기본값 | 사용자의 현재 위치 기준 10km |
| 동적 거리 확장 | 피드 내 글의 수합이 10 미만인 경우, 수합이 10 이상이 될 때까지 탐색 거리를 자동으로 확장한다. 확장 여부는 사용자에게 별도로 표시하지 않는다 |
| 거리 고정 조건 | 10km 이내의 작성글 수합이 10 이상인 경우 탐색 거리는 10km로 고정된다 |
| 정렬 기준 | 1순위: 현재 위치에서 가까운 순. 2순위: 동일 거리권 내에서는 해당 글 작성 위치 기준 10km 이내 최근 수집 활동이 활발한 순 |
| 위치 표시 정보 | 각 글에는 정확한 좌표 대신 동네 또는 장소 라벨과 현재 위치 기준 거리를 함께 표시한다. 예시: `연남동 · 320m`, `서울숲 근처 · 1.2km` |
| 시간 표시 정보 | 각 글에는 작성 시점을 상대시간으로 표시한다. 예시: `방금 전`, `3분 전`, `3시간 전`, `3일 전` |
| 글 표시 정보 | 수집 수, 원작성자, 작성 상대시간 |
| 중복 노출 | 동일한 글은 피드에서 한 번만 노출한다 |

### 7.6 작성자 전용 정보

| 항목 | 내용 |
|-|-|
| 수집한 사람 목록 | 자신이 작성한 글에 한해, 수집한 사람의 목록을 볼 수 있다 |
| 프로필 진입 | 수집한 사람의 프로필에 진입할 수 있다 |
| 접근 제한 | 이 정보는 작성자 본인만 볼 수 있다 |

### 7.7 프로필과 탐색

| 항목 | 내용 |
|-|-|
| 프로필 진입 조건 | 피드에서 글의 작성자 프로필에 진입할 수 있다. 수집 없이 즉시 가능하다 |
| 프로필 내 정보 | 타인의 프로필에서는 그 사람이 작성한 글 피드와 수집한 글 피드를 볼 수 있으며, 각 글에는 작성 상대시간이 함께 표시된다 |
| 프로필 표시 범위 | 프로필은 활성 닉네임과 두 개의 글 피드만으로 구성되며, 소개 문구 등 추가 정보 영역은 없다 |
| 탐색 깊이 | 타인의 프로필에서 발견한 글의 작성자 프로필에 다시 진입할 수 있다. 깊이 제한은 없다 |
| 프로필 개방성 | 프로필 비공개, 잠금, 승인 절차는 존재하지 않는다. 단, 차단 관계에서는 상호 프로필 진입과 콘텐츠 노출이 제한된다 |

### 7.8 사용자 차단

| 항목 | 내용 |
|-|-|
| 지원 목적 | 괴롭힘 방지와 안전 확보를 위한 최소 보호 장치다 |
| 진입 경로 | 타인의 프로필에서 사용자를 차단할 수 있다 |
| 핵심 효과 | 한쪽이라도 상대를 차단하면 두 사용자는 서로의 프로필에 진입할 수 없고, 서로가 작성한 글이 피드와 프로필 탐색에서 노출되지 않는다 |
| 작성자 전용 정보 처리 | 차단 관계인 사용자는 작성자 전용 수집한 사람 목록에서 노출되지 않는다 |

---

## 8. 정책 가정

### 8.1 수정/삭제 규칙

- 글 수정 기능은 MVP에서 제공하지 않는다.
- 작성자는 자신의 글을 삭제할 수 있다.
- 원글이 삭제되면 모든 사람의 수집 목록에서도 제거된다.

### 8.2 위치 정보 노출 수준

사용자에게는 정확한 위도/경도 대신 동네 또는 장소 라벨과 현재 위치로부터의 거리를 함께 노출한다. 예시는 `연남동 · 320m`, `서울숲 근처 · 1.2km` 같은 형태다. 정확한 좌표, 상세 주소, 타 사용자의 실시간 위치는 노출하지 않는다. 시스템 내부에서는 좌표 계산과 반경 판정을 위해 정밀 위치를 사용한다.

### 8.3 차단 정책

차단은 공개 범위 설정 기능이 아니라 안전 장치다. 전체 서비스는 공개 구조를 유지하되, 차단 관계가 발생한 사용자 쌍에 한해 양방향 노출 제한을 적용한다.

## 9. 성공 지표

### 9.1 핵심 지표

- 주간 글 작성 사용자 수
- 주간 수집 사용자 수
- 글 1개당 평균 수집 수
- 작성 후 7일 내 최소 1회 이상 수집된 글 비율
- 프로필 탐색 전환율: 글 열람 후 작성자 프로필로 진입한 비율

### 9.2 품질 지표

- 위치 권한 허용 전환율
- 신고율 및 운영 삭제율
- 도배성 콘텐츠 비율

### 9.3 북극성 가설 지표

"장소에 남겨진 글이 그 장소를 방문한 사람들에게 실제로 수집되는가"를 보여주는 지표로 `글 작성 위치 기준 10km 이내에서 발생한 수집 수 비율`을 추적한다.

## 10. 리스크와 대응

| 리스크 | 설명 | 초기 대응 |
|-|-|-|
| 콜드 스타트 | 초기에는 특정 지역에 글이 거의 없어 피드가 비어 보일 수 있다 | 동적 탐색 거리 확장으로 기술적으로 대응한다. 런칭 지역을 좁히고 초대 기반 밀집 런칭을 검토한다 |
| 스팸/도배 | 위치 기반 서비스 특성상 특정 장소를 반복 점유하는 도배가 발생할 수 있다 | 작성 빈도 제한, 신고, 운영자 비노출 기능을 둔다 |
| 위치 프라이버시 우려 | 정확한 위치 노출은 불안감을 만들 수 있다 | 사용자 노출은 거리/장소 라벨 중심으로 제한한다 |
| 수집 진입 장벽 | 수집의 의미가 직관적이지 않을 수 있다 | 온보딩에서 수집의 의미를 명확히 전달하고, 1차 사용자 반응을 집중 검증한다 |
| 반복 방문 동기 부족 | 알림이 없어 앱을 다시 열 이유가 없을 수 있다 | 이동하면 피드가 바뀌는 경험 자체가 반복 방문 동기임을 온보딩에서 전달한다 |

## 11. 미래 단계에서 검토할 항목

아래는 MVP 검증 이후에만 검토한다.

- 장소 컬렉션 또는 테마 큐레이션
- 특정 지역/이동 경로를 기반으로 한 읽기 모드
- 사진 1장 + 텍스트 300자 형식 지원
- 오프라인 이벤트 또는 공간 파트너십
- 창작자/큐레이터용 통계 고도화
- 수집 수 기반 로컬 랭킹 페이지

## 12. 한 줄 요약

beenhere의 MVP는 "위치에 묶인 글"과 "수집=로컬 신호"라는 두 규칙만으로, 장소에 이야기가 쌓이고 글을 따라 사람이 발견되는지 검증하는 제품이다.
